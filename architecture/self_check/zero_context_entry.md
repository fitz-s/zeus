# Zeus Zero-Context High-Risk Overlay

Use this when a zero-context agent may touch K0/K1, schema, governance, control, lifecycle, canonical truth, DB authority, source/settlement semantics, q authority, live execution, or docs authority routing.

This overlay does not replace default navigation. It adds the high-risk spine.

---

## Minimum Startup Spine

Read directly:

1. `AGENTS.md`
2. `workspace_map.md`
3. `architecture/self_check/authority_index.md`
4. `docs/authority/zeus_current_architecture.md`
5. `docs/authority/zeus_current_delivery.md`
6. `docs/reference/zeus_prediction_market_quant_reference.md`

Then select the matching profile in `architecture/task_boot_profiles.yaml` before using code structure as sufficient context.

Graph/topology output is optional route context. It is never settlement, source, current-fact, or authority-rank proof.

---

## Step 0 — Bound The Task

Do not begin from broad system phrases such as “finish Zeus” or “refactor the system”. Convert broad work into a bounded task with touched surfaces, invariants, and validation.

---

## Step 1 — Classify The Task

Use `architecture/task_boot_profiles.yaml`. Relevant classes include money_path, probability_q, strategy_selection, execution_lifecycle, risk_sizing, data_replay, source_routing, settlement_semantics, day0_monitoring, docs_authority, graph_review, and agent_runtime.

---

## Step 2 — Add High-Risk Reads

For K0/K1/schema/governance/control/truth work, add:

1. `architecture/kernel_manifest.yaml`
2. `architecture/invariants.yaml`
3. `architecture/negative_constraints.yaml`
4. `architecture/db_table_ownership.yaml` when DB/table truth is touched
5. `architecture/money_path_objects.yaml` when economic object/state/side-effect grammar is touched
6. `architecture/docs_registry.yaml` when docs routing is touched
7. scoped `AGENTS.md`
8. focused canonical reference from `docs/reference/AGENTS.md`
9. targeted code/tests

For docs/demotion/archive work, also read `docs/authority/ARCHIVAL_RULES.md` and `docs/archive_registry.md`.

---

## Step 3 — Answer Proof Questions

Before editing, answer:

- What truth surface is authoritative here?
- Which zone is touched?
- Which invariant or negative constraint is in scope?
- Which file owns the behavior?
- Which files are allowed to change?
- Which files are forbidden to change?
- Is this math, architecture, execution, governance, schema, or docs-authority work?
- Which current-fact surfaces are required and are they fresh?
- Which historical/evidence surfaces must remain non-default?
- Which tests/manifests/docs registries must update?

---

## Step 4 — Guardrails

Do not proceed from a starting point that depends on archive/report/evidence/rebuild/packet material as present-tense law, lacks code anchors for q/execution/lifecycle/DB/risk/settlement behavior, invents fallback strategy/unit/lifecycle/q/native-side semantics, treats NO as a casual complement shortcut, uses stale current facts as live truth, creates root-level scratch/plan files, or leaves historical material in default boot.

---

## Step 5 — Evidence Before Merge

A high-risk change is incomplete without targeted tests or honest unavailable status, manifest/invariant references, docs registry/router updates for moved/added/deleted docs, stale-term/default-route checks for docs changes, rollback/demotion notes when authority/reference changes, and clear separation of changed-surface failures from repo-wide drift.

---

## Rule Of Thumb

A zero-context agent should become safe before it becomes expansive. This overlay exists to prevent authority inversion, stale lore resurrection, and live-money semantic shortcuts.
