## Cross-cutting observations

1. **0 NEEDS_FIX is honest, not self-serving.** The 6 NOT_LIVE_PATH classifications fall into 3 categories: pre-SDK constructor failure (Sites 3, 6), pre-SDK preflight (Sites 7, 8), and SDK exception with no dict response (Sites 11, 12). Each category genuinely lacks a `_venue_submission_envelope` to persist, and the persistence helper's contract (`result must be a dict`) at executor.py:446-449 enforces this. The audit is not hiding NEEDS_FIX gaps behind NOT_LIVE_PATH labels — the labels track a real semantic boundary.

2. **Defense-in-depth is genuine.** Site 4/5/9/10 verdicts depend on `final_envelope_payload` being populated by the prior `_persist_final_submission_envelope_payload` call at 1609/2291. If that call raises, control flow diverts to REVIEW_REQUIRED + return BEFORE the SUBMIT_REJECTED branches at 1662/1694/2342/2373 are reached. The `**final_envelope_payload` spread therefore CANNOT execute on an unpersisted result — the typed exception forces an alternative flow.

3. **Test fixtures use live-bound envelopes (T1F dependency satisfied).** Per audit lines 313-321, T1F's `assert_live_submit_bound` gate would block placeholder envelopes before SDK; T1G's tests construct non-placeholder envelopes (token IDs `tok-t1g-aaa...`/`tok-t1g-bbb...`) so the SDK boundary is reached. T1F invariant `T1F-PLACEHOLDER-ENVELOPE-FAKE-SDK-COUNT-ZERO` continues to hold.

4. **No new SDK call surface introduced.** `T1G-NO-NEW-SDK-CALL-PATH` invariant verified: zero source-file diffs, all source files clean. Tests use mocks (no real SDK contact).

## Residual risks (informational, out of T1G scope)

- **Probe 5 LOW**: SDK exception capture at 1538-1548/2222-2232 uses `str(exc)` which depends on the exception's `__str__` implementation. A future hardening could explicitly extract `getattr(exc, 'response', None)` for HTTP errors to capture last-known venue state. Carry-forward to T2F observability hardening.
- **Probe 7 LOW**: No persistence-rate telemetry counter. REVIEW_REQUIRED event fires on failure but no positive counter on success. SRE observability gap, not a security gap. Carry-forward to T2F.
