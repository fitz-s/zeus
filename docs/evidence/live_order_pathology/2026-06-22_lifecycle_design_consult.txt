Recommended realization: FamilyRebalanceIntent, not “lift the same-token invariant”

The superior unification is a single “re-decide held exposure” operation. It should cover three outcomes of the same fresh EDLI redecision: hold/no-op, fill-up same token, or shift-bin by closing the old leg and then possibly entering the new leg. That is simpler and safer than adding separate fill_up, shift_bin, and exit paths because the risk boundary is the same: one family, one held exposure, one fresh decision, one durable lifecycle generation, and no new order unless the current chain/local state still matches the intent.

The input is an EDLI_REDECISION_PENDING event, the complete family topology, fresh executable snapshots, the current q/q_lcb vector, the current position_current row(s), live/pending/unknown venue_commands and order/trade facts, current reserved collateral, and the entry decision certificate for the held position. The existing reactor already treats EDLI redecision as a forecast decision event and threads it through the same q/FDR/Kelly/live submit path; the adapter sets allow_same_family_monitor_owned = event.event_type == "EDLI_REDECISION_PENDING" near the decision body, and the source-truth/phase gates include EDLI redecision in the live boundary. GitHub+1

The state change is a durable row, for example:

family_rebalance_intents(intent_id, event_id, family_key, operation, held_position_id, held_token_id, held_bin_id, selected_token_id, selected_bin_id, q_entry_lcb, q_current_lcb, target_total_exposure_usd, current_exposure_usd, pending_exposure_usd, delta_entry_usd, old_exit_command_id, new_entry_command_id, status, generation, created_at, updated_at, abort_reason, idempotency_key, evidence_json)

Add a unique active index on the family key while status IN ('PLANNED','EXIT_SUBMITTED','EXIT_UNKNOWN','EXIT_PARTIAL','ENTRY_SUBMITTED','ENTRY_UNKNOWN','ENTRY_PARTIAL','REVIEW_REQUIRED'). Precise fill-up condition

Use this as the money-path predicate, not an unscoped same-token bypass:

can_fill_up is true only when all of the following are true.

The event is EDLI_REDECISION_PENDING, the event passed the existing source-truth/phase/fresh snapshot gates, and the fresh decision was produced by the same certified q_lcb/FDR/Kelly route as ordinary entries. The adapter comments say EDLI redecision carries the same FSR-shaped payload and is gated identically, and the reactor’s forecast lane phase gate includes EDLI redecision before scoring. GitHub

There is exactly one open/economically live held exposure for the family, and there are no live/unknown/partial unowned ENTRY commands for the same family. “Exactly one” can allow the current held position_id and no other blocking exposure; if the reducer returns multiple open positions or any unknown command not owned by the same intent, fail closed.

The selected fresh proof is the same token, same bin, same direction as the held position. This is fill-up, not shift-bin. If the fresh primary is a sibling token, route to shift-bin sequencing.

The probability comparison uses the same authority on both sides. Prefer q_lcb_5pct/certified held-side q_lcb, not point posterior, because the money path’s size authority is q_lcb and the attribution file treats immutable decision-time q/q_lcb from the verified certificate as the skill authority. If entry q_lcb is missing, do not fill-up in v1; record FILLUP_ABORT_ENTRY_Q_LCB_MISSING. GitHub

The exact evidence condition should be q_current_lcb > q_entry_lcb plus positive incremental target exposure. If the operator insists on the suggested 0.03 threshold, implement it only as a named evidence hysteresis constant, q_lcb_strengthening_floor = 0.03, with provenance on the receipt; do not present it as a safety cap. My preferred predicate is stricter economically and less arbitrary: target_total_exposure_usd > current_live_and_pending_exposure_usd + venue_min_increment_usd and the incremental cost curve still has positive robust marginal utility. That satisfies “belief strengthened” by letting q_lcb and Kelly/ΔU produce a larger target rather than adding a separate throttle.

The submitted amount is residual: delta_entry_usd = target_total_exposure_usd - current_live_exposure_usd - same_token_pending_entry_usd. Submit nothing if delta_entry_usd <= 0, if it is below venue minimum and cannot be honestly bumped under the existing venue-minimum economics, or if the final submit recapture aborts. This is the critical safety lift: the invariant can be conditionally relaxed only for a residual resize, never for a second full entry.

Shift-bin sequencing

Shift-bin should be close-before-open, not “atomically” in the database sense of holding a SQL transaction across venue calls. The correct atomicity is lifecycle atomicity: the family has one active rebalance lease, and the new entry is impossible until the old exposure is proven closed or dust-only. The README documents the lifecycle phases pending_entry → active → day0_window → pending_exit → economically_closed → settled, and the repo’s risk model distinguishes local/chain state with blocking on uncertain or chain-only facts. GitHub

The sequence should be:

Acquire the family rebalance lease and persist operation='SHIFT_BIN', the old position id/token/bin, and the fresh candidate that triggered the shift.

Re-read exposure and commands. If there is any unowned pending/unknown/partial command in the family, abort with SHIFT_ABORT_BLOCKING_EXPOSURE.

Submit a reduce-only exit sell for the old held token using the existing exit path. The exit executor already builds idempotency from decision_id, token, side, price, size, and intent kind; it checks existing idempotency rows, blocks unresolved economic unknowns, calls can_submit_replacement_sell, and acquires ExitMutex per position/token. GitHub+1

Do not submit the new bin until old residual exposure is confirmed zero or below dust/min-order residual from chain/order/fill facts. If the exit is partial, unknown, review-required, or cancel-unknown, the rebalance stays blocking and no counter-entry is allowed.

After old-leg closure, recompute the full family decision on fresh snapshots/books. Do not reuse the pre-exit selected sibling because the reactor itself has a submit-recapture rule that aborts when family rank reverses and says there is no inline switch/WATCH fallback auto-submit. GitHub+1

If the fresh sibling still wins, submit a new entry sized to the fresh target. If no candidate now passes, end the rebalance as EXIT_ONLY_COMPLETE. That is not a false exit if the exit decision was independently justified and audited; it is a market-moved/no-counter-entry outcome.

Error, boundary, concurrency, rollback cases

If the pre-submit DB write fails before the venue call, no side effect occurred; abort the intent and release the lease. The executor already treats pre-submit database locks as transient reject/no order placed for exits. GitHub

If the venue call boundary is crossed and the outcome is unknown, mark the command and the rebalance UNKNOWN/REVIEW_REQUIRED and block the whole family until reconciliation proves absence or terminal state. Do not attempt a replacement or counter-entry. This is consistent with the executor’s SUBMIT_TIMEOUT_UNKNOWN/SUBMIT_UNKNOWN_SIDE_EFFECT handling and the family exposure blocking phases. GitHub+1

If duplicate EDLI events arrive, they should collide on the family lease or on command idempotency and return the existing intent/command, not create a second order. If an already-live fill-up order exists for the same token, the next EDLI event must see it as pending exposure and either no-op or wait.

If old-leg exit fills but the fresh counter-entry no longer passes, stop. Do not “roll back” the exit by buying the old token again unless a later independent redecision selects it. Venue side effects are not database transactions.

Exit-timing audit design

The existing entry-skill grader should remain entry-skill. It selects terminal positions with phase IN ('settled','economically_closed','admin_closed'), uses the immutable entry decision-q authority, and writes one idempotent settlement_attribution row per position. It also writes settlement-derived PnL to edli_live_profit_audit as (settled_payoff - avg_fill_price) * filled_size - fees, which is explicitly settlement payoff, not exit proceeds. GitHub+2GitHub+2

Add a second attribution axis: exit timing. The counterfactual is:

would_have_settled_value_usd = closed_shares * settlement_payoff_per_share

net_exit_value_usd = exit_proceeds_usd - exit_fees_usd

exit_alpha_usd = net_exit_value_usd - would_have_settled_value_usd

For a YES position sold at 0.85 that later settles 0, exit_alpha = +0.85/share before fees. For a YES sold at 0.20 that later settles 1, exit_alpha = -0.80/share before fees. For partial exits, grade each closed lot by its own closed_shares, leaving remaining shares to settlement/other exit rows.

Categories should be settlement-grounded and separable from entry skill:

SKILLFUL_REVERSAL_EXIT: exit was caused by a model/physics/family-rank reversal against the held side, exit_alpha is materially positive, and final settlement confirms the held leg underperformed the exit proceeds.

LUCKY_EXIT_SAVED_LOSS: exit_alpha is materially positive, but the exit decision certificate did not show q/physics evidence against the held side, or the exit was operational/risk/admin rather than predictive.

PREMATURE_EXIT_COST: exit_alpha is materially negative; holding would have produced more value than selling.

CORRECT_HOLD_THROUGH_EXIT_REJECTED is not needed as an exit category unless a real exit decision was emitted and rejected; do not fabricate exit decisions.

NEUTRAL_EXIT: absolute exit_alpha is within fees/dust/materiality threshold.

ADMIN_OR_RISK_EXIT_VALUE_DELTA: record the dollar delta but exclude from model-skill denominator when exit was forced by admin/risk/cutover rather than predictive evidence.

EXIT_UNATTRIBUTABLE_SETTLEMENT_MISSING: no verified settlement yet.

EXIT_UNATTRIBUTABLE_PROCEEDS_MISSING: settlement exists but exit fill/proceeds are not provable.

EXIT_UNATTRIBUTABLE_Q_MISSING: exit happened, but the exit decision-q/reason certificate is missing; value delta can still be reported, but not counted as model skill.

At exit decision time, capture immutable facts sufficient to attribute the exit to the exit decision rather than the entry: exit_decision_id, position_id, entry_decision_id, family_rebalance_intent_id if any, old token/bin/direction, shares intended to close, q_exit/q_lcb_exit for the held side and selected sibling/family, q authority/provenance hash, trigger reason (EDGE_REVERSED, FAMILY_RANK, physics reversal, risk/admin), current book snapshot id, best bid/ask, limit price, expected proceeds, residual target, command id/idempotency key, and timestamp. At fill time, attach actual closed shares, average exit price, net proceeds, fees, order/trade ids, and residual shares. At settlement time, attach verified settlement outcome, payoff per share, would-have-settled value, exit_alpha, category, and rationale.

Composition without double-counting

Keep two ledgers:

Entry skill answers: “Was the original entry aligned with settlement reality if held?” That remains settlement_attribution.

Exit timing answers: “Given the entry existed, did the exit improve realized value versus the counterfactual hold-to-settlement value for the shares it closed?” That is exit_timing_attribution.

The additive decomposition is:

realized_closed_lot_pnl = exit_proceeds - entry_cost

hold_counterfactual_pnl = settlement_payoff - entry_cost

exit_timing_alpha = exit_proceeds - settlement_payoff

So realized_closed_lot_pnl = hold_counterfactual_pnl + exit_timing_alpha.

This avoids double-counting. A trade can have an entry SKILL_LOSS because the held bin ultimately lost, while the exit row is SKILLFUL_REVERSAL_EXIT because the system sold before the loss. That exactly matches the operator’s statement that a trade aligned initially and then correctly reversed before market notice is also a good trade. Gist

Alternatives weighed and rejected

I reject “just lift the same-token invariant for EDLI when q_current > q_entry + 0.03.” The strongest case for it is minimal code churn and reuse of the existing redecision→entry screen→submit chain. The problem is that the current final stake kernel is not proven residual; it documents a family-total single-leg stake, so a direct lift can submit another target-sized order. It also does not solve duplicate EDLI races or pending unknown commands. GitHub+1

I reject “leave shift-bin as open-new-leg plus old monitor exit later.” The strongest case for it is that it preserves opportunity capture when the new bin is temporarily cheap. The problem is that the code’s own strategy notes say live default is single-leg until a multi-leg execution state machine exists, and family identity is the complete city/date/metric partition, not a venue slug-level distinction. GitHub+1

I reject adding caps, cooldowns, time bans, maker-only, or q haircuts. The operator laws explicitly require honest evidence gates and no artificial caps/throttles/shadow builds, and the current stake kernel already removed an artificial min-order percentage cap for that reason. Gist+1

I reject replacing the entry grader with realized PnL only. It would credit exits but destroy the distinction between entry truth and exit timing. The right model is two composable attribution rows: entry-vs-settlement and exit-proceeds-vs-settlement counterfactual.

Strongest case against my recommendation: close-before-open can miss profitable shifts when the old leg is illiquid or the new bin reprices before the exit fills. That is real EV drag. I still recommend it because the current system does not yet have a tested multi-leg/partial-fill/UNKNOWN state machine, and the binding safety law is “no false exit or entry.” Once a true family portfolio/stake-vector execution engine exists, shift-bin can become a two-leg rebalance with explicit maximum overlap, partial-fill handling, and chain-proven recovery; until then, close-before-open is the minimal safe live implementation.
