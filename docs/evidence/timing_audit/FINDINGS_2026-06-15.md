## TL;DR — the breakthrough thesis

**Zeus is blind to its own timing.** The engine has a fully-designed end-to-end timing
instrumentation schema, and almost none of it is actually populated. The per-hop latency the
operator wants audited (`source_available → fetch → run_complete → forecast → decision →
submit_intent → venue_ack → fill → finality`) **is not recorded anywhere it can be
differenced.** That is *why* every iteration re-tunes the same freshness/backoff/timebox
window constants by guess — there is no measured bottleneck to aim at. The fix path is
**instrument → measure → optimize**, not "re-tune a window."

Second structural finding: the dominant runtime event is **stale/skip**, consistent with the
prior evidence note that cities are *fresh only ~25% of wall-clock time* under a 30s
executable-price window vs a ~120s warm cycle. The binding constraint smells **structural
(window ≪ cycle)**, not any single severed wire. Widening 30s→180s is a band-aid on that.

## C. Why there has been no breakthrough (the reframe)

You cannot optimize what you do not measure. The per-hop latency is unrecorded, so each
session reaches for the only knobs visible in code — the window/backoff/timebox constants —
and tunes them by intuition. The measurements that would say *which hop is the bottleneck*
(decision_events) are dead; the measurements that would say *how stale inputs really are at
use* (quote_age_ms, latency_ms) are null/zero. The loop repeats because the instrument panel
is dark.

## H. Gating unknowns — resolutions (prereq to fix-correctness)

- **U1 (host TZ) RESOLVED:** host runs **America/Chicago** (CDT −0500); the live daemon **inherits it**
  (no `TZ=UTC` in any launch script/plist; running pids show TZ-unset). Now is 20:10 CDT = 01:10 UTC
  (next day) — a live demonstration of the 5–6h window where local date ≠ UTC date.
  **Pervasive-skew fear REFUTED:** naive `datetime.now()` = 2 hits, both comments; `dateutil.parser.parse`
  = 0; `datetime.today()` = 0. The code consistently uses `datetime.now(timezone.utc)` → persisted stamps
  are correct UTC. **Real residual = 10 `date.today()` sites** (return Chicago-local date → off-by-one
  near UTC midnight), which violate the explicit law at `src/contracts/epistemic_context.py:12`
  ("Strictly forbids 'date.today()' scattered locally"). Plus 47 SQLite `datetime('now')` (UTC), coherent
  with the tz-aware Python side. → U1 downgrades the code-map's pervasive-naive-now items to LOW; keeps the
  `date.today()` cluster (WP-12/WP-17/WP-27) as a bounded, enumerable fix prioritized by which sites feed
  `target_date`/settlement-day vs harmless backfill cutoffs.
- **U2 (time_semantics runtime-read) RESOLVED: TEST_ONLY.** `src/contracts/time_semantics.py` is
  imported by no runtime daemon path (only `test_time_semantics_relations.py` + offline tooling). The
  stale `executable_price_freshness_window=30.0` registry entry is **doc-rot, not a live bug** — the
  reactor reads `FRESHNESS_WINDOW_DEFAULT=180s` (executable_market_snapshot.py:47). → **WP-01 downgrades
  to P3 on runtime-correctness**; its real cost is the comments-still-say-30s misleading future sessions
  (rut-perpetuation, P2-process), fixed by registry/comment hygiene.
- **U3 (SDK venue timestamp) RESOLVED: SDK_CARRIES_VENUE_TS, discarded at submit.** WS/REST paths DO
  extract `message["timestamp"]/"matchtime"/"created_at"`, but the submit ACK path (`executor.py:2758`,
  `:3835`) hard-sets `ack_time = datetime.now(timezone.utc)` and reuses it for both `observed_at` and
  `venue_timestamp`, never reading the server ts already present in the threaded `**payload`. → **WP-04/
  WP-11 are FIXABLE** data-quality bugs (server match-time exists, is discarded), not unavoidable.
  Full evidence: `gating_unknowns_2026-06-15.md`.
