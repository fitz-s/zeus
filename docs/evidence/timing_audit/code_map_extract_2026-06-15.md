### Cross-Cutting Themes

- STALE-WITHOUT-DETECTION is the dominant disease, in two forms: (a) a constant/value is widened or updated in one authority but a derived registry/comment still encodes the old value (30s->180s in time_semantics.py + 5 source comments + reactor debounce narrative; reactor cycle budget 30.0 vs '45s' comment); (b) a process-lifetime cache or pin never reloads, so an operator artifact/pin update is silently ignored until daemon restart (EMOS/sigma-floor/mu-offset tables, _PIN_CONFIG_CACHE, sigma_scale_fit candidate). The fix pattern is identical: derive from the single authority, or add a reload/staleness gate + fail-loud.
- MIXED-CLOCK COMPARISON: wallclock event-time vs ingest-write-time vs monotonic vs model-valid-time are compared against each other across joins and gates. recorded_at(write) gates model freshness instead of issue_time(event); expires_at anchored to ingest wallclock not source_cycle_time; entered_at(poll-detect) and venue_timestamp=ack_time(Zeus wallclock) stand in for venue event-time; fit-freshness uses computed_at(ingest) while bucketing uses source_cycle_time(valid); stale_blocked uses monotonic cache-age vs obs event-time age. Each silently biases freshness/grading/calibration by the gap between the two clocks.
- INJECTED decision_time vs LIVE datetime.now() clock-split: the pipeline is built to thread an injected decision_time for replay/causal consistency, but several hot-path sites bypass it with datetime.now() (is_fresh @property, POST-REFRESH RE-ELECT fresh_at=None, executor deadline arithmetic, heartbeat sleep cadence). Benign under low backpressure, dangerous when decision_time lags wall-clock — and the lag magnitude is unmeasured everywhere it matters.
- SILENT-FALLBACK / SILENT-ZERO-ROW: many error and edge paths degrade quietly with no log/alert — ZoneInfo failure -> UTC date off-by-one (no log); lead_days off-by-one -> zero history rows indistinguishable from 'never had history'; forecast_live source_health lock contention -> skip with no metric; season-key mismatch -> None floor; narrow httpx-only exception catch; COALESCE skips absent timing column. A swallowed fallback on a timing axis is the highest-leverage class because it produces wrong-but-plausible numbers, not crashes.
- NAIVE-DATETIME / TZ-OFFSET hazards anchored on an unverified assumption that the SQLite host runs UTC: dateutil.parser.parse boot crash, calibration-store naive-UTC invariant, freshness_gate replace(tzinfo=UTC), date.today() cutoffs, IEM local_valid-as-UTC. Resolving the single 'what TZ is the SQLite/server host?' question collapses or sharply re-ranks ~6 separate audit items.
- UNMEASURED 'GUESS'-BASIS BUDGETS that are load-bearing: executable_price_freshness_window (30s GUESS), snapshot_reserve (12s GUESS), Gamma timebox 2.0s slice below p95 2.516s relying on drain grace, RECONSTRUCTED 12h dissemination lag with no empirical basis, q_lcb bootstrap with no latency budget. These pass review because the code is correct around them, but the magnitude margin to real cost is unknown in every case.
- CYCLE-COUNT-NOT-WALLCLOCK deadlines: void/fault gates keyed on N consecutive cycles (MAX_PENDING_CYCLES_WITHOUT_ORDER_ID=2, _BELIEF_STALE_FAULT_THRESHOLD=3) fire at unpredictable wall-time depending on cycle cadence, with no wall-clock backstop — a stalled cycle loop can hold a position or suppress a fault indefinitely.
- DUAL-USE-ONE-KNOB: a single env var or constant drives two semantically distinct operations with divergent defaults/needs (ZEUS_DISCOVERY_CLOB_TIMEOUT_SECONDS 10s-Gamma vs 5s-CLOB across 5 sites; _WU_STALE_AGE_HOURS duplicating the evaluator constant) — tuning one silently mistunes the other.

- **[MED]** ECMWF 00Z safe_fetch margin is only 5 minutes — provider delay would trigger fallback — forecast_live_daemon fires 00Z download cron at 08:10 UTC vs safe_fetch window opening at 08:05 UTC (00Z + 485 min). Margin = 5 min. Any ECMWF OpenData publication delay beyond 5 min from the expected release causes evaluate_safe_fetch() to return SKIPPED_NOT_RELEASED, triggering fallback to prior cycle. Prior schedule (07:30 UTC) caused fallback regularly. Current +5 min margin is minimal. Suggest monitoring ECMWF actual release times to validate margin.
- **[LOW]** forecast_live source_health deferred 10 min at boot — OpenData health gap on restart — forecast_live_daemon explicitly sets next_run_time = startup + 600s for its source health probe (line 889). On daemon restart, the trading daemon's freshness gate sees stale OpenData last_success_at for up to 10 min from last ingest_main write, not from actual latest check. In a scenario where ingest_main is also restarting, OpenData could appear stale for 10+ min triggering ensemble_disabled.
