# AB3 — Fail-loud lane-write counters + lane-liveness health check

Created: 2026-06-16
Authority basis: timing-semantics fix AB3 (fail-loud lane counters);
docs/evidence/timing_audit/ZEUS_TIMING_COMPLETE_PLAN_2026-06-16.md

## Why
A swallowed decision/telemetry lane-write exception is INDISTINGUISHABLE from a
lane that simply had nothing to write. That blindness let the
`edli_no_submit_receipts` decision lane sit dead from 2026-06-06 with nobody
noticing — the failures were swallowed silently and a dead lane looked
identical to a quiet one. AB3 makes those swallows fail LOUD + COUNTED while
preserving every existing behavior (the `degraded` flag is still set exactly as
before; control flow and swallow semantics are otherwise unchanged; no new
caps/throttles).

## PART 1 — `src/engine/cycle_runtime.py`

### New module-level helpers (after `_increment_summary_counter`)
- `_record_lane_write_failure(summary, lane_name, exc)` — `cycle_runtime.py:2157`
  Increments `summary['lane_write_failures'][lane_name]`, sets
  `summary['degraded'] = True` (preserved), logs
  `logger.error("LANE WRITE FAILED lane=%s err=%r", ...)` (`:2173`).
- `_record_lane_write_success(summary, lane_name)` — `cycle_runtime.py:2176`
  Increments `summary['decision_lane_writes'][lane_name]` (per-cycle success
  count). Does NOT touch `degraded`.

Uses this module's existing module-level `logger` (`cycle_runtime.py:47`).

### Swallow sites converted (file:line -> lane_name)
1. `cycle_runtime.py:4268-4280` — **`_flush_derived_writes`** (the central site).
   This is the single executor every queued derived lane write flows through:
   `opportunity_fact`, `probability_trace`, `microstructure`,
   `forward_market_substrate`, `shadow_signal`,
   `settlement_day_observation_authority`, etc. The `name` arg is the per-lane
   key. On exception -> `_record_lane_write_failure(summary, name, exc)` (`:4276`);
   on success -> `_record_lane_write_success(summary, name)` (`:4280`). This one
   change instruments the ENTIRE derived decision/telemetry-lane family
   (failure-count + per-cycle success-count) at one point.
2. `cycle_runtime.py:5490` — **`shadow_signal_build`**. The shadow-signal payload
   build/queue block previously logged "telemetry write failed" but did not
   count it; now `_record_lane_write_failure(summary, "shadow_signal_build", exc)`.
3. `cycle_runtime.py:4692` — **`executable_snapshot_price_linkage`**. The direct
   (non-queued) `log_executable_snapshot_market_price_linkage` lane write in
   `_capture_execution_snapshot_for_decision`. `forward_market_price_linkage_status`
   = "error" preserved; `degraded` now set via the counted helper.
4. `cycle_runtime.py:~4505` — **`forward_market_substrate`** (`_write_guarded`).
   This guard is itself queued through `_flush_derived_writes`, so its old
   INTERNAL swallow would have made the central handler record a FALSE success
   for the lane. Fix: `_write_guarded` now records its substrate-specific status
   fields (`forward_market_substrate_status`/`_error`) then **re-raises**, so the
   central `_flush_derived_writes` handler is the single place that logs + counts
   the failure (no double-count, no false success). Net behavior identical to the
   prior swallow: `degraded` set, cycle not blocked.

### Per-cycle write map (`decision_lane_writes`)
Implemented at the same central point (`_flush_derived_writes` success branch,
`:4280`): every successfully-flushed derived lane increments its per-cycle
success counter. A lane that reaches neither the success nor the failure branch
(zero writes AND zero failures) is therefore detectable downstream as a possible
dead lane.

### Sites deliberately LEFT (not silent decision/telemetry-lane swallows)
- `cycle_runtime.py:215, 248` — `_source_writer_frontier_status`: sets
  `observability_degraded` in its OWN status dict (not `summary['degraded']`),
  read-only frontier diagnostics. Not a lane write.
- `cycle_runtime.py:4511, 4538` — inside substrate `_write`: STATUS-based degraded
  (`written_with_conflicts` / `skipped_invalid_schema`) on the SUCCESS path of the
  write call (write returned, with a degraded status). Already log warnings; not a
  swallowed exception. (Real exceptions here propagate through `_write_guarded` ->
  central handler.)
- `cycle_runtime.py:4680` — forward price-linkage STATUS-degraded check
  (`_forward_price_linkage_status_degraded`). Success-path status, already counted
  via the direct linkage handler; not a swallow.
- `cycle_runtime.py:4987` — discovery backpressure truncation. Not a lane write;
  already logs a warning.
- `cycle_runtime.py:6294, 6301` — shoulder-exposure ledger failure. This is a REAL
  decision-lane failure but it is NOT silent: it returns an error string (not a
  swallowed exception), already logs a `logger.warning`, AND deliberately triggers
  `entries_paused` / `_freeze_entries_after_shoulder_ledger_failure` control flow.
  Routing it through the counter would risk disturbing that pause control flow, so
  per the additive-only constraint it was LEFT. Flagged here for a future pass if a
  counter is wanted (would need care around the pause semantics).
- `cycle_runtime.py:~4566` — `_write_intraday_depth_guarded`: explicitly fail-soft
  fill-model order-book DEPTH capture; NEVER sets `degraded`; not a decision/
  telemetry lane that gates degraded. Left as-is.

## PART 2 — lane-liveness health check (read-only surface)

### Function placement
`data_lane_health_check(...)` — **`src/control/heartbeat_supervisor.py:624`**
(the prompt's named file). Pure, read-only verdict builder: reads the per-cycle
`lane_write_failures` (lane -> failure count) and `decision_lane_writes`
(lane -> success count) maps, flags (a) any lane with a nonzero failure count and
(b) any lane in an explicit `expected_lanes` set with zero writes, and emits a
`logger.warning` naming each flagged lane. Returns
`{ok, failed_lanes, zero_write_lanes, scope}`. OBSERVABILITY ONLY — it does not
gate or block trading.

### Where it is invoked + WHY there (not in the heartbeat keeper itself)
`src/control/heartbeat_supervisor.py` is the VENUE heartbeat-keeper process
(polls the venue for resting-order safety); it has NO access to the per-cycle
discovery `summary` dict. The cycle summary is surfaced into the cycle-pulse
status file by `cycle_runner.py:_export_status -> write_cycle_pulse(summary)`,
which calls `_refresh_pulse_infrastructure_status(status, cycle_summary)`. That
function already has the summary in hand and already writes structured infra
health into the pulse that `src/control/live_health.py` reads — i.e. it is the
minimal existing channel. So the check is INVOKED from
`src/observability/status_summary.py:261` (inside
`_refresh_pulse_infrastructure_status`), every cycle, and the verdict is surfaced
at `status['data_lane_liveness']` (`status_summary.py:266`). The
`logger.warning` naming any failing lane is emitted by the check itself.

This honors the prompt's escape hatch: "if the per-cycle summary is not reachable
from the supervisor ... implement the check where the summary IS reachable and
note the placement." The function lives in the named supervisor file; it is
called where the data lives.

Note: the call passes `expected_lanes=None` — without an explicit per-cycle
expectation set, a quiet (zero-write) lane is NOT assumed dead (many lanes
legitimately have nothing to write some cycles, e.g. `shadow_signal` flag-gated,
`forward_market_substrate` only when markets exist). The high-value, false-alarm-
free signal is the FAILURE count, which is always surfaced loudly. Zero-write
detection is available to any caller that supplies an explicit expectation. The
import + call are wrapped in a fail-soft `try/except` so liveness surfacing can
never break the pulse. Import direction (`status_summary -> heartbeat_supervisor`)
already exists in the file (`status_summary.py:841`), so no new import cycle.

## NEW TEST
`tests/test_lane_liveness_failloud.py` — 8 unit tests, no engine boot, no live DB.
Covers: (a) `_record_lane_write_failure` increments the per-lane counter, sets
`degraded`, and logs an ERROR naming the lane (caplog); per-lane accumulation;
`_record_lane_write_success` per-lane counting without flipping `degraded`;
(b) `data_lane_health_check` WARNs on a failed lane, WARNs on a zero-write
EXPECTED lane, stays SILENT when all expected lanes wrote, does not flag a quiet
lane without an expectation, and handles `None` inputs gracefully.
