### 4. CENTRAL RISK — TRUST STORED PROOF vs RE-READ VENUE — SOUND (HIGH confidence). Recommendation: TRUST the stored proof; do NOT add a venue re-read.
The stored proof is not a mutable flag — it is an append-only, hash-sealed, guarded-read artifact:
- Generated by `build_absence_proof` (`src/execution/edli_absence_resolver.py:128-175`), which performs a LIVE authenticated CLOB read (`_read_authenticated_venue` -> `get_open_orders()` + `get_trades()`), token-filters, and `raise RuntimeError("authenticated venue read found matching exposure; do not release cap")` (`:173`) if ANY match exists. A proof can ONLY be written when matching_open==0 AND matching_trade==0 at observation time.
- `proof_hash = sha256(proof body)` (`:171`) — the counts/flags the new pass re-reads are exactly what was hashed; they cannot have drifted from the sealed body.
- The top-level `venue_order_exists=False / venue_trade_exists=False / reconcile_reason=AUTHENTICATED_CLOB_ABSENCE... / authenticated_absence_proof=proof` are written in ONE atomic `append_reconciled` under `world_write_lock` (`edli_absence_resolver.py:275-284`). The flags cannot be set true while the embedded proof says absent.
- Soundness argument holds: a submit authenticated-absent at proof time has no resting order and no pending fill on the venue, so it cannot retroactively land; for a NULL-venue_order_id unknown the absence proof stays valid. A venue re-read at resolution time would only re-confirm absence (or find a NEW unrelated order, which it must not attribute to this command). The author's mitigation — re-check both exposure flags + both matching counts + reconcile_reason + link uniqueness + token from the sealed proof — is sufficient defense-in-depth. Re-reading the venue (`edli_absence_resolver._read_authenticated_venue`) is explicitly NOT recommended: the whole reason this pass exists is the live read lane could not complete; coupling resolution to a venue re-read reintroduces the failure mode the fix is meant to survive, and adds a write-while-blocking-on-REST hazard the short-conn refactor (2026-06-11) deliberately removed.
