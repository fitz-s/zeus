## CRITICAL / HIGH findings

### [HIGH, confidence MEDIUM] Fix A FAILS OPEN on `CapTransitioned:PENDING_RECONCILE` — double-submit of a possibly-live order
- Where: `src/engine/event_reactor_adapter.py` — `_LIVE_ORDER_TERMINAL_EVENT_TYPES` (the 4-tuple) and
  `_locked_live_opportunity_active_order_reason` terminal test `event_types & set(_LIVE_ORDER_TERMINAL_EVENT_TYPES)`.
- The defect: the predicate treats the bare PRESENCE of `CapTransitioned` as TERMINAL -> RELEASE,
  ignoring `to_status`. But `CapTransitioned` has three outcomes (producers in
  `event_reactor_adapter.py:5293/5338/5356/5374`):
  - `CONSUMED` (filled), `RELEASED` (closed) — genuinely terminal. OK.
  - `PENDING_RECONCILE` (`event_reactor_adapter.py:5370-5386`) — emitted on submit
    `TIMEOUT_UNKNOWN` / `POST_SUBMIT_UNKNOWN` together with a `SubmitUnknown` event. The live cap is
    held **RESERVED**, and the order **may still be resting LIVE on the venue**. The aggregate/projection
    state machine itself classifies this as non-terminal: `live_order_aggregate.py:266-268` sets
    `current_state="PENDING_RECONCILE"`, `pending_reconcile=True`.
- Consequence: between the timeout-unknown submit and the reconcile loop's closing `Reconciled`
  (`command_recovery.py:5405-5413`, scans `proj.pending_reconcile=1` at `:5726`), the LATEST aggregate
  event is `CapTransitioned:PENDING_RECONCILE`. A re-decision cycle in that window queries the lock and
  is RELEASED -> a second live order on a family whose first order may still be open. This is exactly the
  double-submit the fix's docstring promises to prevent ("FAIL-CLOSED: ... indeterminate ... treat as
  ACTIVE and suppress").
- Reproduced (probe, commit tree): seeding `SubmitPlanBuilt -> ExecutionCommandCreated -> SubmitUnknown ->
  CapTransitioned(to_status=PENDING_RECONCILE)` and calling
  `_locked_live_opportunity_active_order_reason(...)` returns **`None`** (RELEASE), not a suppress reason.
- Not caught downstream: `LiveCapLedger.reserve` (`live_cap.py:78-121`) "caps NOTHING" and dedupes only by
  `(event_id, cap_scope)`; a fresh re-decision carries a new `event_id`, so the cap does not block the
  duplicate. The lock predicate is the load-bearing duplicate guard at this seam
  (`event_reactor_adapter.py:4575/4583` and the selector via `_locked_candidate_no_price_improvement_reason`).
- Fix direction: classify `CapTransitioned` as terminal ONLY for `to_status in {CONSUMED, RELEASED}`;
  treat `to_status=PENDING_RECONCILE` (and the presence of a non-terminal `SubmitUnknown` without a later
  closing `Reconciled`) as ACTIVE -> suppress. Equivalent: prefer the projection `current_state` /
  `pending_reconcile` flag over a bare event-type-set membership test.
- Confidence note: HIGH severity (money-path duplicate live order on the fix's own stated invariant);
  MEDIUM confidence because live reachability depends on a re-decision landing inside the
  submit->reconcile window, which I could not prove occurred in production read-only. The contract is
  nonetheless violated and the test suite does not cover this state, so it blocks.

## MEDIUM / LOW findings

### [LOW, confidence MEDIUM] `Reconciled` treated as terminal even when `pending_reconcile=True`
- `_LIVE_ORDER_TERMINAL_EVENT_TYPES` includes `Reconciled` unconditionally, but `append_reconciled`
  (`live_order_reconcile.py:147-173`) and `rebuild_projection` (`live_order_aggregate.py:269-271`) allow a
  `Reconciled` event carrying `pending_reconcile=True` (matched-pending-finality). The dominant recovery
  path writes `pending_reconcile=False, venue_order_exists=False` (`command_recovery.py:5396-5397`), a real
  terminal closure, and the pending-finality case is a FILL (position, terminal-for-order). So the practical
  exposure is small, but for strict correctness the same `to_status`/`pending_reconcile`-aware classification
  recommended above should also gate `Reconciled`.

### [LOW] Test gap mirrors the HIGH finding
- The rewritten Fix A tests cover ACTIVE-suppress, `Reconciled`-release, SubmitPlanBuilt-only-fail-closed,
  no-prior-order-release, and the SUBMIT_DISABLED `CapTransitioned:RELEASED`-release path — all real
  (they exercise the predicate directly and would fail if the logic were reverted). But there is **no test**
  for the `SubmitUnknown + CapTransitioned:PENDING_RECONCILE` state, i.e. the exact fail-open above. Add a
  RED test asserting suppress for that state alongside the source fix.
