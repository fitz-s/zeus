Representation contract: the target state

The representation layer has three allowed classes.

First, the hand-written kernel. This is small durable law: trading mission, authority order, money-path prohibitions, comment law, naming law, metadata law, and the query protocol. It lives in root AGENTS.md plus tiny scoped AGENTS.md files. It must be token-budgeted and checked for size and forbidden stale patterns.

Second, checked policy inputs. These are small structured YAML files that cannot be generated because they encode operator law or ontology: invariant registry, negative constraints, canonical vocabulary, and representation contract. Each record must name its checker or generated evidence. No prose-only record is valid.

Third, generated/queryable surfaces. Module indexes, call graphs, DB read/write ownership, deprecation indexes, test coverage maps, public API summaries, import topology, scar candidates, and boot cards are generated into a single queryable artifact, preferably SQLite plus JSONL exports for cheap agent ingestion.

Everything else is deleted, archived outside boot, or converted.

The hard rule is:

A representation surface may exist only if writer != none and drift_detector != none.

For hand-written law, the writer is the human/operator and the drift detector is a schema plus semantic checker. For generated metadata, the writer is a generator and the drift detector is deterministic regeneration plus diff. For code comments, the writer is the code author and the drift detector is a comment-law linter plus ID validation. If a surface cannot name both, it is a lie waiting to happen.

Comment law

The proposed comment law is right but should be tightened.

A comment or docstring may exist only when it contains at least one non-derivable fact in one of these categories:

World fact: units, venue semantics, WMO/HKO/DST behavior, settlement preimage, clock boundary, external API ambiguity, rate-limit behavior, market convention.

Invariant reference: a stable ID such as INV-17, INV-28, NC-13, with the comment explaining only the local binding, not restating the invariant.

Why-not fact: why the obvious implementation is wrong, especially where the code intentionally violates a common idiom.

Proof/provenance rule: rank order of truth sources, not an anecdote about a past incident.

Concurrency/ordering rule: commit-before-export, idempotency boundary, crash-recovery sequence, terminal-state caveat, side-effect ordering.

Safety/rollback rule: what must remain true during partial failure, restart, or operator abort.

Everything else is banned: restating control flow, narrating what the code does, dated incident history, “created/last reused/audited” headers, local TODO lore, module-purpose paragraphs, authorial explanations, and comments that duplicate CodeGraph queries.

The pinned code has both good and bad specimens. command_recovery.py contains comments/docstrings that often state non-derivable semantics: a SQL reducer ranks proof strength before recency; terminal venue status alone is not enough because cancel/expire can follow partial fills; explicit matched size must not be inferred. Those are load-bearing semantic facts and should survive, but with invariant IDs and a standard shape. By contrast, replacement_forecast_materializer.py has a module header with created/reused/audited dates, “FIX” incident narration, and long operational story comments. The semantic facts inside it may be real, but the incident narrative belongs in receipts/history/git, not inline code. GitHub+2GitHub+2

The required inline format should be small and parseable:

# repr: kind=world_fact ids=[INV-SETTLE-ROUND] owner=contracts.settlement

The next line may contain the fact, capped at one sentence:

# unit=degC; venue settlement rounds official preimage, not forecast display value.

For why-not comments:

# repr: kind=why_not ids=[INV-28,INV-30] rejects=terminal_status_implies_zero_fill

# Cancel/expire after partial fill is terminal but not proof of zero matched size.

For SQL fragments or other places decorators cannot attach:

# repr: kind=proof_order ids=[INV-28] artifact=venue_command_truth_cte

# Proof rank dominates recency; weaker later facts cannot demote stronger earlier truth.

Docstrings have a stricter law. Public exported functions/classes may have a one-line contract if it is not derivable from the type signature. Private functions get no docstring unless they bind a non-derivable fact. Multi-line prose is justified only for concurrency, safety, or external-world semantics that cannot be expressed in types, schema, or invariant IDs. The target docstring form is:

Python

Run

def canonical_order_truth(...) -> OrderTruth:
    """Return proof-ranked order truth, not latest venue echo. repr: ids=[INV-28,INV-30]"""

For a private helper in a well-named module, the ideal is often no docstring at all. A typed signature plus canonical names should carry the local meaning; the invariant registry and event schema carry the why.

Enforcement should be implemented as scripts/check_repr_contract.py --comments, integrated into the existing doctor/linter path rather than creating a new governance stack. It should fail changed files when:

A comment/docstring lacks a repr: tag and is not a mechanical pragma such as # noqa, # type: ignore, shebang, encoding declaration, or generated-code marker.

A comment contains date-like incident lore, FIX, HACK, created, last_reviewed, audited, reused, authority basis, or “incident” outside a receipt/test fixture.

A repr: tag names a missing invariant, negative constraint, vocabulary concept, schema, or owner.

A docstring for a private function merely repeats the function name, arguments, or return type.

Legacy files should be advisory at first. New or touched code should be blocking immediately.

Naming law

Shorter names are not automatically better for LLM agents. The optimum is the shortest identifier that preserves the canonical domain concept and its authority plane. A name that saves tokens but collapses settlement, grade, observation, and venue echo is expensive because the agent will infer the wrong ontology. A name that repeats the module path wastes tokens and dilutes attention.

The rule is:

One canonical token per domain concept. No synonym unless the vocabulary registry declares it as an alias for migration.

Do not repeat package/module context inside local identifiers.

Prefer strong domain nouns over generic verbs.

Use abbreviations only when they are canonical domain notation, not convenience compression.

Encode authority plane when collision risk is real: served_q, venue_order, settlement_grade, forecast_snapshot, decision_receipt.

Do not encode implementation trivia, temporal incident history, or module echo.

The repo already documents vocabulary collisions: replacement_forecast has aliases, Day0 is overloaded, family is overloaded, and the glossary names synonym families such as shadow/diagnostic/legacy, ghost/unsubmitted/abandoned, runtime_posture/regime/mode, and substrate/book/venue. These are not style issues; they are ontology bugs for an agent. GitHub

At the import boundary, the caller should say replacement_forecast.materialize(...) or import a domain-named module alias. Module echo is expensive and adds no information.

Concurrency and ordering constraints: representation generation must be deterministic, must not read live trading state, must run after schema/code generation and before CI comparison, and must serialize edits to canonical vocabulary and invariant IDs. DB/export invariants should preserve the existing commit-before-derived-export law. Parallel rebuild waves can modify code freely, but vocabulary and invariant registry changes require a single merge owner per wave. GitHub+1

Boundary cases: dynamic SQL needs explicit repr:dynamic_sql plus a contract test; generated code is exempt only if the generator is registered; temporary packet docs are excluded from boot and expire on closeout; notebooks and ad hoc scripts cannot be authority; external API semantics must enter as world-fact invariants or schema fields, not comments.

Leverage and cuts

Highest leverage per unit effort: comment law on changed files. It immediately stops the deadliest failure mode: plausible stale prose in the code path. It is cheap to lint and rides every rebuild wave.

Second: event/certificate schemas plus DB access contracts. Agents need intent-flow and authority ownership more than module summaries. A schema-bound decision receipt or venue command contract is vastly more useful than prose saying “this module handles decisions.”

Third: canonical vocabulary. This pays off because the repo has live ontology collisions: q/posterior/belief, receipt/certificate, settle/grade, family, regime/runtime_posture/mode, and multiple packages that sound like decision authorities. GitHub+1

Fourth: generated module index and boot cards. Valuable, but less urgent than stopping stale comments and authority-plane ambiguity.

Cut as over-engineering: a new ADR bureaucracy, per-file YAML sidecars, natural-language module summarizers, full retroactive comment cleanup across untouched legacy code, and a separate “representation governance doctor.” Extend the existing doctor/linter system instead. The repo already has enough governance surfaces; the rebuild should reduce entities, not add a fourth generation.
