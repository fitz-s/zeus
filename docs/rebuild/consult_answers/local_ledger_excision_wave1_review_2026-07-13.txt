What is missing before LX-2R may start

The first missing prerequisite is a truthful LX-0R capability fence. The current packet must stop advertising ACTIVE_NEW, the canonical economic identities and unit/rounding contracts must be frozen, and the generated manifest must be paired with runtime access inventory. A reducer interface should not be frozen while the source ownership vocabulary is still incomplete.

The second is a replay-grade continuous source spine. Payouts need complete hash-pinned vector generations, canonical/finality status, and registry-driven historical coverage. Fills need raw retention of Zeus, foreign, and ambiguous observations; exact fees and provider revisions; durable alias edges; real pagination/source cursors; order-sync coverage; and a watermark that proves everything returned was persisted. Balances need monotonic collateral and exact CTF token observations, the complete token-discovery history, and explicit reservation-to-observed-commitment handoff identities.

The third is identity supersession. Historical and ongoing duplicate-position relations must be represented as immutable POSITION_IDENTITY_SUPERSEDED facts, with a backfill completeness/ambiguity report and a coverage marker the reducer can fail closed on. No historical position read-model backfill is trustworthy while that input is absent.

The fourth is completion of LX-E rather than its current archival approximation. New ENTRY commands must have mandatory exact certificate attribution; the EDLI path must append canonical fill facts; execution-quality and settlement-learning outputs must be independently versioned immutable receipts; and the historical learning corpus must be backfilled and checked, with ambiguous associations marked rather than guessed. The Round-2 adjudication expressly places this world-database learning migration before LX-2R. GitHub

The fifth is an actual integrated branch and operational source-health proof. All required synchronizers must be enabled, their cursors and freshness visible, and the combined schema/tests must run together. The packet SHAs alone do not prove that schemas, schedulers, settings, or imports coexist at one deployable tip.

Selector migration, mixed-savepoint separation, dual-capable recovery/reconcile/exit branches, dormant database firewalls, generation publication, full replay, and cent-equivalence are then LX-2R work and acceptance gates. They do not excuse starting the formal phase with incomplete LX-1R inputs. Isolated reducer implementation can be prototyped, but it must not establish a supposedly stable input contract, run an acceptance replay, or be represented as phase progression until the prerequisites above are fixed.

LX-F revert: prerequisite or parallel work?

LX-F rework is a hard prerequisite to LX-2R historical replay and sign-off. It is not necessarily a blocker to writing isolated reducer code in parallel.

The dependency boundary is precise: reducer developers may implement a versioned supersession input and unit-test it with synthetic graphs while LX-F is reworked. They may not backfill the production read model, claim cent-equivalence, or publish a complete generation until the immutable supersession facts and their coverage report exist. The adjudicated plan specifically requires duplicate consolidation to become identity-supersession facts before read-model backfill because otherwise duplicate position identities can be counted or attributed independently. GitHub

The reverted implementation should not simply be re-landed unchanged. Its commit changed schema/event definitions and backfill behavior, but the supplied change set does not show a live-database migration for an already-created SQLite event-type CHECK constraint. The rework needs an explicit compatible schema migration, stale-binary fencing, an idempotent backfill, and a report for consolidations whose absorbed identities cannot be proven. I found no public reason in af902a8e4 beyond the mechanical revert message, so I do not infer why the operator reverted it. GitHub+2GitHub+2

The strongest counterargument is that a globally unique economic-fill reducer could prevent dollar double-counting without any position supersession: if every fill contributes once, duplicate position shells need not duplicate economics. That would be viable only if the target model never consumes legacy position identity for phase, command linkage, settlement, certificate attribution, reviews, or portfolio publication. The adjudicated four-contract design and the landed certificate-attribution work are position-linked, so that condition is not currently true. A complete redesign around command/fill identity could change this dependency, but it would itself require a new adjudication and proof.

Correctness, security, migration, and rollback assessment

The most important security property in LX-T1 passes: the observer has no signing authority and does not submit redemption transactions. I found no new direct SQL-injection surface in the reviewed parameterized inserts. The material security risks are integrity and authority risks instead: a build that over-advertises ACTIVE_NEW, a watermark that certifies discarded observations, and source events that label an order quote as a fill price can all admit an incorrect economic authority without compromising credentials.

Concurrency hazards remain in payout RPCs performed during a database write transaction, non-monotonic wallet-head UPSERTs, and mutable receipt supersession without an independently versioned identity. Ordering hazards include same-value payout reorgs, fee-only trade corrections, late/backdated fills after a wall-clock watermark, and certificate conflicts silently ignored by position-level DO NOTHING.

Most new tables are additive, so code rollback is mechanically possible. Data rollback must not mean deleting or editing observations that were already treated as immutable. Bad payout/fill observations should be superseded; untrustworthy source watermarks should be invalidated and replayed; stale heads should remain auditable observations while a newer canonical head is selected; and derived read models should be dropped and rebuilt. LX-F rework additionally needs an explicit migration path for existing SQLite schemas and a compatibility window for stale binaries.

NO-GO for beginning LX-2R as an adjudicated execution phase, running production/history replay, freezing the reducer input contract, or claiming cent-equivalence.
