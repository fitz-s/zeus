2. CORE CORRECTNESS

The core formulation is sound if implemented as a strict separation between belief, executable venue state, and utility. q must be the only settlement belief; fills, price, and book state have utility value but no settlement-information authority. This is exactly the order-engine design’s A1/A3 split: q is a MECE family vector, while order actions are FOK/FAK, post-only maker, cancel, convert/split/merge, sell, and no-op decisions under venue constraints. GitHub

The load-bearing mathematical correction is to delete downstream q-derived gates and make the solver consume q directly. The q violation ledger states that q point vectors were structurally correct while downstream q_lcb, NO-leg modeling, edge, sizing, and exit logic were broken; its largest root cause was independent per-bin percentile treatment of raw non-renormalized bin-mass samples, which collapses high-belief lower bounds and poisons entry decisions. GitHub

The right decision objective is expected utility under one bankroll, not filter-then-rank. The implementation doc explicitly says the current family decision engine is a filter-then-rank top-1 chooser and that the missing piece is a joint convex/global solver. GitHub The superior realization is: enumerate a discrete action menu per family, compute executable payoff vectors against settlement bins and route costs, then solve for the bankroll allocation that maximizes expected log utility or a closely related concave utility with one explicit risk knob κ. For small menus, brute-force/discrete repair around a convex relaxation is acceptable; for dependent families, scenario-based optimization dominates independent per-family water-filling.

The hardest correctness boundary is cross-family dependence. Daily temperature bins across nearby cities and same model cycles are not independent. A first-principles solver needs a scenario service: either calibrated residual scenario samples, a shrinkage covariance/copula model, or a conservative dependence envelope. The implementation doc allows an interim single-family solve with caps but requires stamping the correlation rail as caps; that is acceptable only as a migration rail, not as the final architecture. GitHub

The maker-order formulation is under-specified by the operator axioms. If historical fill intensities and price-conditioned belief are both disallowed, a posted maker order cannot be optimally valued in the usual expected-fill sense. The minimal safe interpretation is adversarial/conservative: a maker quote is allowed only when it remains acceptable under worst-case fill timing until the next freshness boundary, cancel latency, and self-trade/rate-limit constraints. Otherwise the solver should prefer taker actions or no-op. This is a real premise challenge: banning learned fill intensity removes a useful execution model; it may be worth the simplicity, but it makes maker optimization deliberately conservative rather than fully optimal.

The settlement/grading boundary is correct and non-negotiable. Settlement values must enter only through SettlementSemantics; grading must bind the frozen decision certificate to the eventual settled outcome; calibration must consume only walk-forward-eligible settled outcomes. The current authority doc treats settlement semantics, feed roles, high/low metric identity, runtime truth ordering, and fail-closed behavior as money-path laws rather than style preferences. GitHub

The most likely wrong input class is a stale or partially ordered event set: new model cycle arrives, old resting order remains live, partial fill occurs, cancellation is delayed, and the solver sees a mixed q/book/ledger snapshot. The implementation doc correctly requires the reservation ledger before event cadence, because event-driven triggering amplifies the ledger race. GitHub Therefore W1/K0 ledger and order-state reduction are correctness prerequisites, not cleanup.

Security-wise, the minimal architecture is safer because only the executor can touch venue credentials and only certified commands can reach the executor. Operator ARM and RiskGuard remain the live submit boundary. The current architecture already documents OperatorArm as live-submit authorization in the daemon, and the implementation doc classifies ARM/RiskGuard as reuse kernels with a deletion trap around coverage gates. GitHub+1

3. VALIDATION SOUNDNESS

The current evidence is not yet a sound demonstration that the rebuilt system will make money or preserve invariants live. The design docs are strong architectural evidence, but they are not proof of q calibration, solver optimality, ledger linearizability, or live venue parity. The settlement-graded EV document explicitly caveats that it is an offline pass rather than a full live sizing/exposure replay because exact historical state was not reconstructable, and its graded after-cost EV was sign-indeterminate: mean −0.0026 with 95% CI [−0.0380, 0.0321] over 269 decisions. GitHub

The single load-bearing claim still asserted but not proven is:

“The Probability Authority produces an honest, timely posterior q that is positive after executable costs when acted through the new live solver under one bankroll.”

To prove that claim soundly, validation must become certificate-native:

Every live or replayed decision must have a frozen certificate containing q_version, input HWMs, book snapshot id, ledger snapshot id, solver menu, selected plan, and reason for no-op.

Replay must be event-log deterministic under duplicate, reordered, missing, and late events.

Ledger tests must be adversarial: concurrent reserve/fill/cancel/settle/convert operations must preserve the bankroll identity.

Solver tests must include brute-force small-market comparisons, monotonicity checks, stale-q rejection, market-closed/no-liquidity boundaries, partial-fill prefixes, self-trade prevention, rate-limit exhaustion, and Day0 absorbing cases.

Settlement tests must cover high/low dual-track identity, HK truncation, WMO half-up, final source role, and settlement-source unavailability.

The existing docs acknowledge many of these gates—q tripwires, latency SLA, ledger identity, self-trade guards, REST_ELIGIBLE, and byte-identical OFF promotion gates—but the evidence is not yet the same as an adversarial verifier suite. GitHub+1

Top 5 structural defects a from-scratch design would never contain

Economics re-proved in multiple layers. q admission, edge, sizing, floor gates, selection correction, strategy lanes, maker escalation, and exits each partially decide whether the trade is good. A first-principles design has one solver.

Recovery used as substitute for canonical truth. A from-scratch system would first define order/ledger state and then make recovery a reducer over venue facts, not a 16K-line compensating subsystem.

Scan/cycle orchestration as the primary money-path trigger. The irreducible trigger is source/book/fill/schedule event freshness; scanning is a watchdog.

Semantic primitives not consistently enforced by type boundaries. Weather settlement systems cannot tolerate ad hoc high/low, rounding, source-role, or float-price semantics. The authority doc correctly treats these as catastrophic category errors. GitHub

Large god files as authority boundaries. event_reactor_adapter.py, command_recovery.py, state/db.py, and main.py are too large to make invariants locally checkable. A minimal design makes truth ownership small enough that a verifier can exhaust its boundary.

Premise challenges

No shadow modes worsens validation. It reduces the ability to compare a new solver against live conditions without risk. Acceptable substitute: event replay plus byte-identical OFF evidence gates; weaker substitute: pure unit tests.

No history-derived caps is good for avoiding performance overfit, but bad if it forbids operational latency bounds. Cancel p99, submit p99, websocket blind-window duration, and model-cycle arrival distributions are structural venue/system measurements, not alpha caps. The order-engine design already uses latency SLA and cancel-rest-time reasoning, so this axiom should be interpreted narrowly. GitHub

q-only belief is correct for avoiding self-referential market feedback, but it can be alpha-negative if the book reflects exogenous weather information before local ingest. The compromise is: market price never updates q, but q/book divergence becomes diagnostic, priority, and validation evidence.

Fail-closed freshness must distinguish entries from exits/cancels.
