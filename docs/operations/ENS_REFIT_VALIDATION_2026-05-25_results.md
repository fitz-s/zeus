## 3-bin Hand-Check: RPS and PIT

**Case:** p = [0.5, 0.3, 0.2], Y = 0 (first bin actual)

**F(j) = cumsum(p)** = [0.5, 0.8, 1.0]

**RPS** = Σ_j (F(j) − 1{Y≤j})²
  - j=0: (0.5 − 1)² = 0.2500
  - j=1: (0.8 − 1)² = 0.0400
  - j=2: (1.0 − 1)² = 0.0000
  - **RPS = 0.2900**  ← harness returns 0.2900 ✓

**PIT** u_i = F(Y) = F(0) = **0.5000**  ← harness returns 0.5000 ✓

**Second case:** p = [0.2, 0.5, 0.3], Y = 1 (middle bin)
  F = [0.2, 0.7, 1.0]; 1{Y≤j} = [0, 1, 1]
  - j=0: (0.2−0)²=0.04; j=1: (0.7−1)²=0.09; j=2: (1.0−1)²=0
  - **RPS = 0.1300** ✓;  **PIT = F(1) = 0.7000** ✓

## Hong Kong Anomaly Note

§4.1 shows HK full_transport_v1 Brier=1.1570 / LogLoss=10.09 vs raw(none) Brier=0.9774.

**Cause: non-overlapping date ranges, not a model blowup.**
- `none` family: 2024-01-01 to 2026-02-28 (winter-dominated, cold outcomes).
- `full_transport_v1` family: 2024-03-01 to 2026-05-03 (summer-dominated, warm outcomes).

The model mass peaks at 33–35°C (summer bins) while outcomes cluster at 28–32°C
— a systematic warm-bias in the transport correction for HK, but the comparison to
the 'none' family is apples-to-oranges (different seasons entirely).

**Gate implication:** HK is NOT a fold artifact. It is a real per-city transport issue
surfaced only on summer dates. The full.db run (which covers both season ranges for
both families) will give the authoritative city-level comparison.
