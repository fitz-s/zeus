## 6. Critical implications for the backtest design

### 6.A. F11 antibody on TIGGE-sourced data ŌåÆ `RECORDED` tier, not `DERIVED_FROM_DISSEMINATION`

TIGGE JSON files already carry `issue_time_utc` extracted from the GRIB header ŌĆö that IS the authoritative issue time, not a reconstruction. Therefore when Zeus ingests these files:

```python
# When the TIGGE writer populates ensemble_snapshots_v2 from these JSON files:
availability_provenance = AvailabilityProvenance.RECORDED  # NOT derived/reconstructed
issue_time = parse_iso(json_doc["issue_time_utc"])
manifest_hash = json_doc["manifest_sha256"]                # F18 antibody satisfied
training_allowed = json_doc["training_allowed"]            # already pre-stamped
```

The 6h40m dissemination-derivation formula in 01 ┬¦5 only applies to **other** forecast sources where Zeus's `forecasts` writer doesn't carry source-header issue_time. TIGGE is the ideal authoritative path.

### 6.B. Provenance fields ALREADY in the source data

- `manifest_sha256` ŌåÆ satisfies F18 (untagged source) and F16 (INSERT OR REPLACE without hash check)
- `issue_time_utc` ŌåÆ satisfies F11 (reconstructed available_at)
- `training_allowed` ŌåÆ already pre-labeled, NOT silently defaulting to true
- `missing_members` ŌåÆ transparency on incomplete ENS (51 members may not all be present every run)
- `local_day_window` ŌåÆ DST-aware aggregation window already encoded

**Implication.** The forensic P1 provenance hardening work for the TIGGE ingest path is dramatically simpler than the WU writer remediation ŌĆö TIGGE already meets P1 contracts at source. The writer just needs to STAMP these fields onto `ensemble_snapshots_v2` rows on insert.

### 6.C. LOW track structurally present at source (data-side; writer-side still missing)

Both `tigge_ecmwf_ens_mx2t6_localday_max/` and `tigge_ecmwf_ens_mn2t6_localday_min/` exist at scale on the VM. C4/F3 (no LOW track) is purely a Zeus *writer-side* gap; the upstream data is ready.
