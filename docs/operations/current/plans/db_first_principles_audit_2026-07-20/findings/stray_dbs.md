### 1. `zeus.db` (1,073,152 B, 74 tables + 157 indexes) — **QUARANTINED**
- **Provenance**: `src/state/db.py:68` — `ZEUS_DB_PATH = STATE_DIR / "zeus.db"  # LEGACY — remove after Phase 4`. It carries the full pre-K1-split monolithic schema (all 74 tables that today live split across `zeus-world.db`/`zeus-forecasts.db`/`zeus_trades.db` — `position_current`, `execution_fact`, `settlements`, `observations`, `trade_decisions`, `risk_actions`, `venue_commands`, etc.).
- **Data check (safe full-count, file is 1MB)**: every data table is **0 rows**. `zeus_meta` has exactly 1 row: `('observation_data_version', 'v0', '2026-05-20 20:21:42')` — matching the file's mtime (May 20 16:21 local). This is a **schema-only artifact from a single `init_schema()` run on 2026-05-20**, never populated.
- **DANGEROUS question — is any live path resolving here today?** Traced the two production connection getters:
  - `src/engine/cycle_runner.py:get_connection()` (the live daemon's trade+world connection, used by `exit_lifecycle.py`, `post_trade_capital.py`, `fill_tracker.py` via `deps.get_connection`) → `connect_or_degrade(_zeus_trade_db_path(), write_class="live")` → resolves to canonical `zeus_trades.db`, then ATTACHes `zeus-world.db`/`zeus-forecasts.db`. **Never touches `zeus.db`.**
  - `src/riskguard/riskguard.py:_get_runtime_trade_connection()` (line 173-177) → `get_trade_connection_with_world_required(write_class="live")` in production (`get_connection.__module__ == "src.state.db"` is always true outside tests). The bare `return get_connection()` fallback on line 176 only fires under test monkeypatching. **Riskguard's live tick never touches `zeus.db`** despite the stale module docstring claiming it does.
  - `src/state/portfolio.py:2839` `get_connection(path.parent / "zeus.db", write_class="live")` is a fallback branch gated by `if trade_db.exists()` (checked first) — since `zeus_trades.db` (94GB) always exists, this branch is dead in practice.
  - **However**: `src/state/db.py:get_connection(db_path=None, ...)` defaults `db_path = db_path or ZEUS_DB_PATH` (db.py:1683) and is **fail-open** (creates the file if missing, `mode` implicit rwc). ~15 call sites invoke it bare with no path argument — all are manual/offline scripts, none are launchd/cron-scheduled (checked every `~/Library/LaunchAgents/com.zeus.*.plist`, no `StartInterval`/`StartCalendarInterval` agent invokes any of them): `scripts/baseline_experiment.py:251`, `audit_replay_fidelity.py:22`, `etl_historical_forecasts.py:62`, `validate_dynamic_alpha.py:140`, `backfill_cluster_taxonomy.py:37`, `backfill_ens.py:161`, `investigate_ecmwf_bias.py:27`, `etl_asos_wu_offset.py:22`, `automation_analysis.py:27`, `run_replay.py:366`, `etl_solar_times.py:83`, `data_completeness_audit.py:60`, `audit_time_semantics.py:20`, `capture_replay_artifact.py:35`, `force_lifecycle.py:13`. Anyone running one of these by hand today gets a **silent near-empty-DB read/write** against the schema-only `zeus.db`, not the 40-94GB canonical files — matching the GPT-consult "fail-open rwc connect" finding concretely.
- **Verdict**: `zeus.db` is inert for the live 24/7 trading path (verified by code, not by trusting the docstring) but is a live hazard for manual script invocation. QUARANTINED — do not delete blind (Phase-4 removal is explicitly still pending per the code comment; some of the 15 callers may still intend to target it), but flag every bare-`get_connection()` call site above for an explicit-path audit before Phase 4 closes.
