# Single-live semantics extinction

Status: EXECUTING
Owner: operator-directed session
Class: architecture + governance + schema vocabulary
Money path: source truth -> forecast signal -> decision -> execution -> monitoring

## Problem

Zeus is live-only, but active source, schemas, automation, and authority prose still
carry `observation`, `telemetry`, probe, veto-only, rehearsal, and equivalent
non-executing parallel-path concepts. Some are merely misleading names; others are
live inputs or installed jobs whose labels deny their real runtime role. Those
concepts create a recurrent design affordance for dormant parallel systems.

## Existing authority

- Root `AGENTS.md` §2: observation modes are extirpated; live may act and backtest may
  evaluate.
- INV-05: risk is behavioral; advisory-only risk is forbidden.
- `docs/authority/replacement_final_form_2026_06_09.md`: live replacement rows use
  `runtime_layer='live'`.
- `docs/authority/zeus_current_architecture.md`: canonical truth and derived
  evidence are separate planes.

The operator's 2026-07-22 instruction explicitly approves deletion, demotion,
renaming, and removal of active-law surfaces that preserve the forbidden concept.

## Current evidence

- Canonical `forecast_posteriors` has only `runtime_layer='live'` rows.
- The same forecasts DB still stamps live materializer inputs as `OBSERVATION_ONLY` or
  `TELEMETRY_ONLY` and retains legacy tables named for the removed mechanism.
- launchd still registers `com.zeus.calibration-transfer-eval`; local LaunchAgent
  backups retain `ZEUS_OPPORTUNITY_BOOK_OBSERVATION`.
- Active source still exposes telemetry scheduler/source roles, a replacement
  forecast live dry-run module, and live receipt fields marked telemetry-only.

## Required shape

1. One live runtime path. A fact is executable, blocked, historical evidence, or
   offline counterfactual input; no dormant live mode exists.
2. Data objects express their own validity, completeness, and runtime layer. They
   do not carry fake trade-authority labels when they are ordinary live inputs.
3. Health checks, audits, receipts, and offline replay are named for their actual
   function. They do not form a second decision system and cannot authorize live
   behavior.
4. Installed jobs are either required live/post-trade work with truthful names and
   effects, or they are removed from source recipes and the host.
5. Active authority, registries, tests, and generated topology contain a blocking
   relapse antibody; historical evidence is not a default routing surface.

The relapse antibody inspects executable live surfaces and current
authority/reference/router surfaces. It rejects the retired concept families and
their variants, not merely a closed list of old identifiers. Fail-closed
rejection and offline replay remain valid only as direct behavior with typed
boundaries; neither may become a selectable alternate live runtime.

## Execution waves

### W0 — census and reachability

- Classify every active-tree hit by runtime reachability and semantic role.
- Prove current process/config/DB/job state before any live-state mutation.
- Record delete/rename/retain decisions; retention requires a precise non-parallel
  runtime role.

### W1 — live code and schema source

- Delete dead switch/rehearsal/probe/veto machinery.
- Remove non-authoritative status columns/enums from live data objects where the
  field cannot change a valid action.
- Rename required health/audit/receipt fields to their actual function.
- Remove dormant branches rather than translating their names.

### W2 — automation and host residue

- Remove obsolete launchd recipes and scheduler registrations.
- After source and tests prove removal, unload/delete the exact obsolete host job
  and stale plist backups; do not touch the active live-trading plist.

### W3 — authority and relapse prevention

- Rewrite active authority/reference/router language to state one-live semantics
  directly.
- Add a blocking active-tree semantic/token gate with narrow exclusions for
  unavoidable external names or historical archive interfaces.
- Delete or archive non-load-bearing active evidence that advertises the removed
  architecture.

### W4 — proof

- Run affected unit/relationship tests and schema migration tests.
- Run source/script/test/docs topology checks and planning-lock check.
- Re-scan active tracked files and host launchd state.
- Obtain independent verifier review of the final diff; fix or disposition every
  finding before closeout.

## Acceptance criteria

- No live-reachable branch, status, flag, scheduler role, table, or installed job
  represents a disabled/observe-only parallel decision mechanism.
- No current live DB input is labeled as non-live while being consumed by the live
  materializer or decision path.
- Offline replay and read-only audits cannot write canonical live truth or authorize
  a live action, and their names state replay/audit rather than a latent live mode.
- Active authority and routing surfaces cannot recommend reintroducing the removed
  concept.
- A blocking test rejects future reintroduction in active runtime/governance paths.
- Current runtime state is re-sampled after every authorized host/DB mutation.

## Safety boundaries

- Work only in the linked branch worktree; never edit or commit directly on `live`.
- No venue order, cancellation, risk posture change, daemon restart, or live DB
  mutation is part of W0/W1 source work.
- Any live DB migration must be explicit, transactional, schema-checked, and run
  only after code lands; no default full-DB copy is permitted.
- Preserve unrelated concurrent worktrees and dirty state.
