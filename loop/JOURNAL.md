# Loop Journal (append-only)

Who writes: `loop/tick.sh` (L1, hourly) and `loop/daily.sh` (L2, daily) —
each invocation appends exactly one block after it finishes, plus mechanical
`VIOLATION:`/`ESCALATION:`/`FALLBACK` lines from `scripts/ops/loop_guard.py`
when the wrapper's own safety checks fire. Entries are never edited or
rewritten after being appended.

Who reads: the next tick (orients from the tail before doing anything —
see `loop/prompts/l1.md`/`l2.md` ORIENT step), the operator (the morning
report is generated by grepping this file, there is no separate digest
file), and L2's own next run (reads the last `cursor:` line to resume the
settlement join where it left off).

**If this file silently stops growing, the loop is dead.** Check
`launchctl list | grep com.zeus.loop`, then `loop/HALT`, then
`loop/logs/tick-*.log` for the most recent run — in that order.

## Format spec

One block per tick, oldest first:

```
## <ISO8601 UTC timestamp> <L1|L2> tick
queue_item: <ledger id | "none">
action: <one-line what happened>
verifier: PASS | FAIL | N/A
commit: <sha | "none">
notes: <optional free text>
```

An empty tick (no new evidence, nothing queued) still gets one line so a gap
in this file is diagnostic, not ambiguous:

```
## <timestamp> L1 tick — empty (no new evidence, queue empty)
```

Grep-able marker lines (appear standalone, inside or after a tick block):

- `VIOLATION: <detail>` — the post-run allowlist diff check
  (`scripts/ops/loop_guard.py enforce`) found a change outside
  `loop/allowlist_auto.txt`; the offending path(s) were hard-restored to
  HEAD and are not part of the tick's real output.
- `ESCALATION: <detail>` — the diff circuit breaker tripped (>20 files or
  >600 changed lines in one tick — ALL new-this-tick changes were
  hard-restored, not just the offending ones) or a queue item failed 3
  consecutive ticks. Needs operator eyes; non-zero ESCALATION count is the
  headline of the next L2 morning report (design doc §3).
- `DEVIATION: <detail>` — the tick took an action outside brief
  expectations but within tier rules; logged for audit, not necessarily a
  problem.
- `cursor: <value>` — L2's `settlement_skill_attribution` join cursor (last
  processed rowid). Read by the next L2 tick only; L1 ignores it.
- `## <date> L2 morning report` — L2's daily summary block: settlement
  evidence processed, ledger status changes, queued PREPARE branches (if
  any), VIOLATION/ESCALATION counts since the prior report, new cursor.
